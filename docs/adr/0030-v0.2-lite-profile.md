# ADR-0030: v0.2 Lite Profile

## Status

Proposed

## Date

2026-05-20

## Context

ADR-0001 through ADR-0029 describe a production-grade system. After a Phase 0 review, the v0.2 target is reframed as a single-user local-machine deployment. Many concurrency / multi-tenancy / scale concerns in the original ADRs do not apply at this scale. This ADR specifies a Lite profile that captures the minimum runnable shape and explicitly defers heavier subsystems to a later Standard profile (Phase 2+).

## Decision

### Profile boundary

v0.2 ships the Lite profile. The original 27 ADRs remain as the design reference for the Standard profile that v0.3+ will move toward.

### Deployment model (Lite)

```
Developer's machine
└── Single Python process (the daemon)
    ├── CLI (Typer entry point)
    ├── FastAPI HTTP server on 127.0.0.1:8080
    ├── ReAct Agent (asyncio single event loop)
    ├── Local SQLite database file at ~/.ai-coding-cli/state.db
    ├── Operation logs as Markdown files in {workspace}/docs/operations/
    ├── Jira / GitHub API clients (polling-only, no webhook receiver)
    └── Local Web Dashboard (read-only HTMX pages)
```

One developer, one workspace, one daemon process, one SQLite file. No external services beyond the company LLM endpoint, Jira, and GitHub.

### What's OUT for Lite (deferred to Standard profile)

| Subsystem | Reason for deferral |
|---|---|

| Neo4j graph integration | Graph queries are not required for the single-project, single-developer workflow |

| Memory four-layer architecture | Replaced by simple single-layer cache: last N operation logs are searchable; no Working/Episodic/Semantic separation |
| Memory Governance pipeline | No conflict detection, no aging — single user owns their own facts |

### Storage Layer (Lite)

Single SQLite database at `~/.ai-coding-cli/state.db` with extensions:

- `sqlite-vec` for vector search (loaded as a runtime extension)
- WAL journaling for concurrent reads while a write is in flight

Tables (mirrors a subset of ADR-0019 schema, with foreign keys and indexes):

```sql
sessions
conversations           (→ sessions)
turns                   (→ conversations)
operation_logs_index    (frontmatter cache; full body lives in Markdown file on disk)
rag_chunks              (text + embedding)
processed_jira_events   (idempotency)
skill_invocations       (audit)
config_snapshots
```

`memory_entries`, `memory_governance_log`, `neo4j_outbox` from ADR-0019 are not created in Lite.

Migrations via Alembic still apply; one migration sequence covers the Lite schema.

### Embedding (Lite)

Single embedding model configured in `.env`:

```
LLM_EMBEDDING__MODEL_NAME=text-embedding-3-small
LLM_EMBEDDING__BASE_URL=...
LLM_EMBEDDING__API_KEY=...
```

Embeddings are computed synchronously when an operation log is written or a RAG chunk is indexed. No background worker. For single-user scale (~10s of operations per stage), inline embedding is fast enough.

### Pipeline simplifications

Single-project means:

- `ProjectRouter.route()` always returns the configured workspace; cross-project detection is bypassed.
- `CrossProjectDesignHandler` and contract validation are not instantiated.
- Sub-ticket fan-out at design approval is not invoked.

Jira polling instead of webhook means:

- `Neo4jSyncWorker` not started.
- `WebhookReactor` becomes a `PollingReactor` only.
- Polling interval default 60s for active tickets, 300s for idle (same as ADR-0029).

### Compactor (Lite simplification)

When a Conversation's `messages_json` exceeds the configured token budget (default ~120k of 200k window), trigger a single-shot LLM summarization that compresses the middle portion:

```python
async def compact_simple(messages: list[Message], llm: LLMAdapter) -> list[Message]:
    # Keep system + first user message + last 5 turns intact.
    # Summarize everything in between with one LLM call.
    # Replace the middle with a single [COMPACTED_SUMMARY] system message.
```

No MicroCompact / AutoCompact distinction. No memory extraction hook (no Memory subsystem). The compacted summary stays in the conversation messages.

### Guardrail (Lite simplification)

Input layer:

- Regex-based prompt injection detection (no LLM-based check).
- Patterns: `^(System|<\|im_start\|>system)`, `"ignore all previous instructions"`, basic credential shapes (`sk-...`, `ghp_...`, etc.).
- Score ≥ threshold (default 0.8) blocks input.

Output layer:

- Regex-based secret leak detection.
- Sensitive-files-list (workspace's `.ai-coding-cli/sensitive-files.txt`): if the LLM quotes content from a matching file, rewrite to `[REDACTED]`.

Action layer:

- Destructive-class tools (per ADR-0013 `SideEffectClass`) trigger CLI confirmation prompt.
- Non-destructive writes (LOCAL_WRITE / EXTERNAL_WRITE) proceed without confirmation.
- Confirmation timeout 5 minutes → auto-refuse.

### Configuration (Lite)

`.env.example` lists ~10 required + ~5 optional fields:

```bash
# Required (8)
WORKSPACE_PATH=/path/to/your/sandbox/repo
JIRA_BASE_URL=https://jira.your-company.com
JIRA_AUTH_KIND=pat
JIRA_API_TOKEN=...
GITHUB_BASE_URL=https://github.your-company.com/api/v3
GITHUB_TOKEN=...
LLM_PRIMARY__BASE_URL=https://llm.your-company.com/v1
LLM_PRIMARY__API_KEY=...
LLM_PRIMARY__MODEL_NAME=gpt-4o

# Optional with sensible defaults
LLM_EMBEDDING__MODEL_NAME=text-embedding-3-small  # (default)
LLM_EMBEDDING__BASE_URL=                           # (default: same as LLM_PRIMARY)
DAEMON_HTTP_PORT=8080
AGENT_MAX_TURNS=20
LOG_LEVEL=INFO
```

A `ai-coding init` command scaffolds `.env`, `.ai-coding-cli/conventions.md`, and the workspace directories per ADR-0016 §`ai-coding init`.

### Concurrency model

Single asyncio event loop. No multi-process, no shared memory across processes. Within the loop:

- One Agent.run() at a time per Conversation.
- Multiple Conversations in different Sessions can run concurrently if the developer kicks off multiple stages, but each is in its own asyncio Task on the same loop.
- Database writes are serialized by Python (single connection in v0.2; pool for parallel reads only).
- SQLite WAL allows concurrent reads while a write is in progress.

No row-level locks needed. No outbox sync race. No webhook delivery race (polling is one place, the polling loop). The "race condition" findings from the review do not apply to Lite.

### Failure modes still in scope

Even at single-user scale, these failure modes are real and must be handled:

- LLM provider down for > 1 hour → detect after N consecutive failures, log structured error, halt cleanly.
- API key expired → detect via 401 response, log credential failure, halt with clear message.
- Daemon crash mid-conversation → on next start, mark `running` conversations as `failed` with reason "daemon_crashed".
- Operation log file vs SQLite row inconsistency → on startup, scan `docs/operations/` and reconcile against SQLite index.
- SQLite file corruption → manual `ai-coding storage backup` + restore documented; corruption detected via `PRAGMA integrity_check` on startup.

The original Top 10 review findings 3, 4, 5, 6, 9, 10 all still apply to Lite. Items 1, 2, 7, 8 are reduced or eliminated because the underlying complexity is gone.

### Upgrade path to Standard profile

When v0.3+ adopts the Standard profile:

1. Add PostgreSQL + pgvector; migrate SQLite data via a one-time `ai-coding upgrade-to-postgres` command.
2. Optionally add Neo4j + outbox sync per ADR-0022.
3. Replace the simple cache with the four-layer Memory architecture per ADR-0020 and Memory Governance per ADR-0023.
4. Add LLM-based Guardrails per ADR-0025.
5. Add webhook receiver + rate limiting per ADR-0029.
6. Enable MCP bridge per ADR-0013.
7. Enable cross-project routing per ADR-0006.

Each upgrade is a separate version increment; users can stay on Lite if they don't need the heavier features.

### Phase 1 of Lite

Implementation order (target 3-4 weeks for a working v0.2 lite end-to-end):

1. Week 1: SQLite schema + migrations + config + LLM Adapter + Tool Registry + native Jira/GitHub tools.
2. Week 2: Agent Core ReAct loop + Context Layer + simple Compactor + Session/Conversation persistence.
3. Week 3: Pipeline orchestrator + Stage 1 (Design) flow + operation log writer + Jira polling reactor.
4. Week 4: CLI commands + local Web Dashboard (read-only views) + Skill Loader + Lite Guardrails.

After Week 4, an end-to-end run on a real Jira ticket validates the Lite profile.

## Consequences

- v0.2 ships as a single Python package with one SQLite file as its only external state.
- One developer can `pip install ai-coding-cli`, `cp .env.example .env`, fill 8 fields, run `ai-coding daemon start`, and have a working pipeline within minutes.
- Architecture remains compatible with the Standard profile: the data shapes (Session / Conversation / OperationLog) are the same; storage changes are an upgrade, not a rewrite.
- Many concurrency / governance / multi-tenancy concerns from the original review are eliminated at the source rather than mitigated.
- Standard profile features (Memory, Graph, MCP, cross-project) remain documented; teams that need them adopt v0.3+.
